# Workshop Design: Teach Through Necessity, Not Features

**Date**: 2026-01-05
**Source**: Workshop brainstorm session - context-finder → Build Your Oracle reframe

---

## What We Learned

**Core insight**: The best workshops teach outcomes, not implementations. Students should discover technical patterns through necessity, not be told about them as features.

### The Reframe

**Before**: "Workshop: Learn the context-finder pattern"
- Focus: Technical implementation
- Hook: "Save 95% on API costs"
- Risk: Students learn pattern but forget why it matters

**After**: "Workshop: Build Your Own Oracle"
- Focus: Outcome (knowledge system that survives)
- Hook: "Build an AI that remembers everything forever"
- Discovery: context-finder emerges as necessary solution to death spiral

### The Emotional Journey

Students must FEEL the problem before discovering the solution:

1. **Week 1**: Build Oracle (excitement - "It remembers everything!")
2. **Week 3**: Add more knowledge (success - "Getting smarter!")
3. **Week 4**: Notice rising costs (concern - "Getting expensive...")
4. **Week 5**: Hit the wall (crisis - "$5 per search, can't afford my own brain")
5. **Week 6**: Despair (giving up - "Maybe I should stop using Oracle")
6. **THE MOMENT**: Discovery (aha! - "What if cheaper model finds, expensive model understands?")
7. **Week 7**: Relief (survival - "Oracle can live forever now")

**Key**: Students discover context-finder because they NEED it, not because you TOLD them about it

## How Things Connect

### Pattern Recognition

This connects to several Oracle principles:

1. **Patterns Over Intentions**
   - Don't declare what the pattern is
   - Let students discover it through behavior (their own pain)
   - Discovery → deeper retention

2. **External Brain, Not Command**
   - Workshop doesn't command: "You must use context-finder"
   - Workshop mirrors: "Your Oracle is dying. What will you do?"
   - Students solve their own problem

3. **The Death Spiral as Teacher**
   - Month 1: $75/month (manageable)
   - Year 2: $3,000/month (unsustainable)
   - Without experiencing this arc, students won't internalize WHY

### Universal Teaching Principle

This applies beyond Oracle:

| Traditional Teaching | Discovery Teaching |
|---------------------|-------------------|
| "Here's the pattern" | Build first, hit wall |
| "This saves money" | Experience the cost pain |
| "Use this technique" | Discover solution yourself |
| Forgotten in weeks | Retained for years |

**Examples**:
- Database indexing: Let queries slow down → discover indexes
- Git branching: Mess up main → discover feature branches
- Testing: Ship bugs → discover why tests matter

**The principle**: Pain → Discovery → Retention

## Application to Workshop Design

### Workshop Structure

**Day 1**: Build Oracle (no mention of context-finder)
**Week 1-4**: Use Oracle, let costs rise naturally
**Day 2 Morning**: "Oracle is dying" (students face crisis)
**Day 2 Afternoon**: Two-phase search insight (students discover context-finder)
**Day 3**: Implement survival mechanism

### What NOT to Do

❌ "Today we'll learn about context-finder, a delegation pattern"
❌ "Here's how to save 95% on costs"
❌ "Let me show you the right way to architect this"

### What TO Do

✅ "Build Oracle. Use it for a week. Come back and tell me what happened."
✅ "Your search cost $5. What does this mean for Year 2?"
✅ "What are your options when Oracle becomes too expensive?"
✅ Let students propose solutions (they'll invent context-finder)

## Real-World Evidence

**From Nat's Agents evolution**:
- context-finder wasn't designed upfront
- It emerged from necessity (system was getting expensive/slow)
- 18 git commits show gradual discovery
- Now powers 50% of slash command infrastructure

**The retrospectives prove it**:
- Early: "Main agent reads everything" (naive)
- Middle: "This is burning context tokens" (pain)
- Later: "Delegate to Haiku for gathering" (discovery)
- Now: "context-finder is core infrastructure" (retention)

**Same journey students should take** in compressed time (3 days vs 3 months)

## Metrics for Success

### Workshop Success = Students Discover

**Not**: "I learned about context-finder pattern"
**But**: "I discovered that Oracle needs indexed access or it dies"

**Not**: "Here's how to save costs"
**But**: "I felt my Oracle dying and figured out how to save it"

**Not**: "Good workshop, learned some techniques"
**But**: "This changed how I think about building systems that scale"

## Next Steps for Workshop Development

When designing curriculum:

1. **Plan the breaking points**
   - Where will students hit the wall?
   - What pain will they feel?
   - How do we compress 3 months into 3 days?

2. **Resist the urge to rescue**
   - Don't jump in with "here's the solution"
   - Let them struggle for hours (not weeks)
   - Guide discovery, don't provide answers

3. **Document the emotional arc**
   - Not just "teach X in hour Y"
   - But "students feel Z, discover X, understand Y"

4. **Test with beta cohort**
   - Did they discover on their own?
   - Or did we tell them?
   - If we told them → redesign that section
