ARCHITECTURE DECISIONS & LESSONS LEARNED
═════════════════════════════════════════════════════════════════════════════════

DECISION 1: Archive → Restore Cycle (Jan 11)
─────────────────────────────────────────────
WHY: Needed to clean up and reorganize skill infrastructure
HOW: Mass archive, then selective restoration
RESULT: Successful recovery in 2 hours
LESSON: Skills can be quickly restored if properly documented

DECISION 2: Skill Symlink Architecture (Jan 13-14)
───────────────────────────────────────────────────
WHY: Need for central skill management across multiple repos
HOW: Move skills to oracle-proof-of-concept-skills, symlink locally
WHEN: Announced Jan 14 20:33 (932c51c8)
RISK: Merge conflict on Jan 15 00:21 (resolved by choosing main)
LESSON: Symlink migration needs clear communication + conflict resolution

DECISION 3: Template Standardization (Jan 15)
──────────────────────────────────────────────
WHY: Skills had inconsistent YAML frontmatter
HOW: Discovered via plugin discovery failures
FIX: Made templates explicit with proper headers
IMPACT: Prevents future parsing errors

KEY PATTERNS IDENTIFIED
───────────────────────
1. Progressive Disclosure
   - Skills introduced gradually
   - Each iteration adds features
   - Users adapt to changes over time

2. Tight Coupling Risk
   - /watch coupled to browser extension
   - /learn coupled to /project skill
   - Needs cleaner separation of concerns

3. DuckDB as Query Engine
   - Replaces grep-based approaches
   - Used in /schedule, /md
   - Emerging pattern for structured queries

4. Symlink Complexity
   - Plugin discovery depends on symlinks
   - YAML parsing depends on frontmatter
   - Need for clear migration path

5. Merge Conflict Pattern
   - Deprecation decisions cause conflicts
   - Solution: Take non-deprecation version
   - Implication: Slower deprecation needed
